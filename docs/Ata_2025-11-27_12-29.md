# Ata da Reunião de Planejamento da Sprint 3
## Desenvolvimento do Frontend Web

**Assunto:** Definição estratégica e técnica para a criação de uma interface web própria para a lista de compras, em substituição ao Notion.
**Participantes:**
*   Alpha (Líder/Mediador)
*   Architect (Arquiteto de Software)
*   Experience (Designer de Experiência)
*   Builder (Desenvolvedor)

---

### Pauta da Reunião

1.  **Mudança de Escopo:** Análise do feedback do usuário final e a decisão de cancelar o uso do Notion.
2.  **Definição da Sprint 3:** Alinhar a nova missão de desenvolver um frontend web integrado ao Flask.
3.  **Planejamento Técnico:** Definição da stack tecnológica, estrutura de diretórios e fluxo de dados.
4.  **Planejamento de Experiência:** Desenho da interface principal com foco em usabilidade mobile.
5.  **Validação e Execução:** Confirmação do plano e ordem para implementação.

---

### Resumo das Deliberações

A reunião foi iniciada por **Alpha** às 12:02, comunicando uma mudança de escopo crucial: o cancelamento do Notion como interface, motivado pela necessidade de uma solução mais simples e direta para o usuário final. A missão da **Sprint 3** foi oficialmente definida como o desenvolvimento de um frontend web próprio.

1.  **Definição da Arquitetura (Architect, 12:03):**
    *   A stack tecnológica **Flask + Jinja2 + Bootstrap** foi validada como a mais pragmática para garantir velocidade e responsividade.
    *   A estrutura de pastas padrão do Flask foi confirmada: `src/templates/` para arquivos HTML e `src/static/` para CSS/JS.
    *   Foi definido o fluxo de dados para a página principal: a rota `GET /` no `app.py` será responsável por consultar o banco, agrupar os itens por categoria em um dicionário Python e passá-lo para o template Jinja2.

2.  **Design da Experiência do Usuário (Experience, 12:18):**
    *   A interface foi projetada com foco total em **mobile-first**, apresentando uma lista vertical única.
    *   Os itens serão agrupados visualmente por categoria, cada uma com um título (`<h2 class="category-title">`).
    *   A solução "No-Zoom" para marcar itens foi detalhada: usar um `<label>` estilizado com dimensões mínimas de `44px x 44px` como alvo de toque, escondendo o `<input type="checkbox">` padrão.
    *   Foram especificadas as classes CSS essenciais (`.product-card`, `.product-info`, `.product-checkbox`) para a implementação.

3.  **Validação Técnica e Contrato de Dados (Architect, 12:18):**
    *   O Architect validou o design do Experience, confirmando que ele se alinha aos princípios de desacoplamento e eficiência.
    *   Foi estabelecido um **"contrato de dados"** preciso: a estrutura que o backend deve entregar ao frontend (um dicionário com chaves de categoria contendo uma lista de objetos de item, cada um com `id`, `nome` e `detalhes`).

4.  **Confirmação e Execução (Builder, 12:18 - 12:23):**
    *   O Builder confirmou ter recebido todas as informações necessárias do Architect e do Experience, sem ambiguidades para iniciar o trabalho.
    *   Alpha emitiu a ordem para o Builder executar o plano.
    *   Às 12:23, o Builder reportou a conclusão da tarefa.

---

### Decisões Tomadas

1.  **Cancelado** o uso do Notion como frontend do projeto.
2.  **Aprovada** a criação de um frontend web próprio como foco da Sprint 3.
3.  **Aprovada** a stack técnica: Flask, Templates Jinja2 e Bootstrap.
4.  **Aprovado** o design da interface mobile-first, com agrupamento por categorias e checkboxes grandes para fácil interação tátil.
5.  **Definido** o contrato de dados que a rota `GET /` deve fornecer ao template.
